# AgentOS Agent Run Preflight Workbench 上机题

## 1. 题目背景

AgentOS 是一个企业级 Agent 运行平台。用户请求会先被拆解成多个子任务，再由 Agent 调用 Skill、Tool、Sandbox 等能力执行。

在企业场景下，AgentRun 真正执行前必须经过一次 Preflight 预检，用于判断：

- 执行计划是否结构合法。
- Skill / Tool 是否可用。
- 依赖关系是否存在环路。
- Token 预算是否足够。
- 是否需要审批。
- 是否需要 Sandbox，以及是否需要生产环境升级。
- 并发计划是否超过租户上限，是否需要改写成串行执行。

请实现一个全栈小功能：**Agent Run Preflight Workbench**。

该功能面向平台操作员和研发调试人员，用于在执行前检查 AgentRun 计划，输出阻断原因、审批原因、预算估算、Sandbox 路由建议、warning、审计说明和 normalized plan。

## 2. 时间要求

建议完成时间：**3-4 小时**。

允许使用 AI Coding / Codex。评估重点不是代码量，而是：

- 需求分析是否清楚。
- 技术设计是否合理。
- 后端规则建模是否稳定。
- 前后端边界是否清晰。
- 测试是否覆盖关键风险。
- 是否能在有限时间内做出可运行、可评审的工程切片。

## 6. 后端业务规则

必须实现以下规则。

### 6.1 Skill 规则

- 任务引用不存在的 Skill：阻断，`decision=blocked`。
- Skill `status != active`：阻断。
- 任一 Skill `action_tier=T3`：不直接执行，需要审批。

### 6.2 Tool 规则

- 任务引用不存在的 Tool：阻断。
- Tool `status != active`：阻断。
- Tool `exposure_mode=deny`：阻断。
- Tool `exposure_mode=ask`：不阻断，但需要审批。
- Tool `exposure_mode=allow`：可继续执行。

### 6.3 DAG 依赖规则

- 任务依赖不存在的 task：阻断。
- 任务依赖形成环：阻断，并返回 cycle path。
- 无环且依赖完整：通过。

### 6.4 Token 预算规则

- 所有任务的 `estimated_tokens` 求和。
- 总和超过 `remaining_token_budget`：阻断。
- 未超过预算：返回 `remaining_after_preflight`。

### 6.5 并发改写规则

- 同一 `parallel_group` 内任务数超过 `max_parallel_tasks`：不阻断。
- 需要在 `normalized_plan` 中把超额任务改写为串行依赖。
- 返回 warning，说明哪个 parallel group 被改写。

示例：

```txt
max_parallel_tasks = 2
parallel_group g1 = [t1, t2, t3]
```

可以改写为：

```txt
t1, t2 保持并行
t3 depends_on 增加 t2
```

具体改写策略可自行设计，但必须稳定、可解释、可测试。

### 6.6 Sandbox 规则

- Skill 的 `sandbox_level` 可为：`none / L0 / L1 / L2 / L3`。
- `none` 表示不需要 Sandbox。
- 非 `none` 表示需要 Sandbox route。
- `environment=prod` 时，`L0/L1` 必须自动提升为 `L2`。
- Sandbox 升级不阻断，但必须返回 warning 和 route reason。

### 6.7 决策优先级

最终 decision 只能是以下三种之一：

```txt
blocked
requires_approval
ready
```

优先级：

1. 有任何 errors：`decision=blocked`, `can_execute=false`。
2. 无 errors，但需要审批：`decision=requires_approval`, `can_execute=false`。
3. 无 errors 且无需审批：`decision=ready`, `can_execute=true`。

## 7. 前端要求

新增页面：

```txt
/admin/runtime/agent-run-preflight
```

### 7.1 页面功能

页面需要包含：

- JSON 输入区，预置一个示例请求。
- 参数区：
  - environment: `dev / staging / prod`
  - remaining token budget
  - max parallel tasks
- 操作按钮：
  - Load sample
  - Validate
  - Reset
- 结果区：
  - 总状态 badge：`ready / requires approval / blocked`
  - Budget summary
  - Governance summary
  - Sandbox routes
  - Errors
  - Warnings
  - Audit notes
  - Normalized plan JSON

### 7.2 前端状态要求

必须处理：

- empty：尚未执行预检。
- editing：用户正在编辑 JSON 或参数。
- loading：请求进行中。
- result：展示成功响应。
- error：展示请求失败或本地校验失败。

### 7.3 前端约束

- invalid JSON 时必须前端本地拦截，不调用后端。
- 前端请求逻辑要封装成 client 或 hook。
- 不要把请求、解析、展示、状态管理全部塞进一个超大页面组件。
- UI 不要求复杂视觉，但需要可读、可操作、状态清楚。

## 10. 加分项

- 规则被拆成清晰 checker / policy，而不是一个巨大 if-else。
- `normalized_plan` 能标记每个改写原因。
